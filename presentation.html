02 · Same text, different representations

\n

Training Pipeline

\n
\n
\n
\n
\n

Controlled comparison

\n

The same FAQ dataset is split with an 80/20 stratified train-test workflow.

\n
\n
\n

Shared preprocessing

\n

Lowercasing, unicode accent stripping, regex tokenization, and stopword removal (scikit-learn ENGLISH_STOP_WORDS).

\n
\n
\n

Two feature routes

\n

Naive Bayes uses word counts; Logistic Regression and Linear SVM use TF-IDF unigram and bigram features.

\n
\n
\n
\n
\n
NB
\n
DataPreprocessingWord CountsNaive Bayes TrainingPrediction
\n
\n
\n

\n
DataPreprocessingTF-IDF FeaturesLogistic RegressionPrediction
\n

SVM
\n
DataPreprocessingTF-IDF FeaturesLinear SVM TrainingPrediction
\n
\n
\n
\n
\n

Naive Bayes — formula-drivenP(c) = Nc/N · P(w|c) = (count+1)/(Nwords+|V|). No vectorizer — classification score computed entirely from log-probabilities.

\n

Logistic Regression — TF-IDF vectorsText converted to weighted numeric features. TF rewards frequent words; IDF penalises words common across all categories. Enables a linear boundary to separate classes.

\n

Linear SVM — TF-IDF + max marginSame TF-IDF features as LR. SVM finds the hyperplane that maximises separation between classes — particularly effective on high-dimensional sparse text vectors.

03 · Why macro F1 matters

\n

Evaluation Method

\n
\n
\n
\n \n
\n
\n
\n

The long-tail problem

\n

A few categories hold most of the questions while many have only a handful. A model can look strong on overall accuracy by getting the big classes right, yet quietly fail the small tail classes, such as fees or IT accounts, that students still ask about.

\n
\n

How we measure model quality

\n
\n Report accuracy and macro F1 together\n Macro F1 weights every class equally\n Per-class precision, recall, and F1\n Train-test gap to catch overfitting\n Stratified 5-fold CV grouped by FAQ id\n Confidence fallback when scores are low\n

01 · Build independent evidence

Three Retrieval Scores

\n
\n
01

Sentence embedding similarity

Encodes the whole user question and compares it with pre-computed FAQ question embeddings. Unit-normalized embeddings make the dot product behave like cosine similarity.

sembed(i) = q · di= cos(q, di)q: user question · di: FAQ question i
\n
02

TF-IDF sentence similarity

Transforms the question into a TF-IDF vector and compares it with all FAQ questions. This captures exact words, short phrases, and lexical anchors.

stfidf(i) = vq · vi||vq|| ||vi||Cosine similarity over TF-IDF vectors
\n
03

Token-level embedding similarity

Removes stopwords, compares every query token with candidate question + keyword tokens, then averages each token's best semantic match.

stoken(i) = 1m Σa=1m best(ta, Ti)best(ta, Ti) = maxb cos(e(ta), e(cb))Average each query token's best candidate-token match
\n

RRF
Score Fusion

retrieval.py uses Reciprocal Rank Fusion, so each method votes by ranking instead of contributing raw scores on incompatible scales.

If one FAQ stays near the top in semantic, lexical, and token rankings, it receives a high final score.

\n

rank 2
\n

rank 1

\n
normalizeRRF(i) = RRFraw(i)3 / (K + 1)
\n
constantK = 60

\n
